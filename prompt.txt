Software Requirements Specification (SRS) for CampusRide – University Carpooling Application Version 1.0 Date: November 1, 2025 Prepared by: Team Delta
INTRODUCTION

1.1 Purpose This document defines the requirements for CampusRide, an application that allows university students and staff to share rides to and from campus, reducing transportation costs and environmental impact.
1.2 Scope CampusRide will be available as both a web-based and mobile application. Users can offer or request rides, schedule recurring trips, and pay drivers via the in-app wallet.
1.3 Definitions, Acronyms, and Abbreviations Driver: A user who offers a ride. Passenger: A user who books a ride. Trip: A planned journey between two points. Wallet: A digital balance for in-app transactions.

OVERALL DESCRIPTION

2.1 Product Perspective CampusRide will integrate with the university’s existing transport management system to verify user identities.
2.2 Product Functions The system shall:
Allow users to register and verify their identity using a university email address.
Allow drivers to post rides and passengers to book seats.
Handle payments through an in-app wallet linked to bank cards.
Send live GPS updates of ongoing trips.
Generate monthly reports for the university administration.
2.3 User Classes and Characteristics Students: Primary users; mainly access via mobile phones. Staff: Occasional users; access mainly via desktop web app.
2.4 Operating Environment The mobile app will run on Android 10+ and iOS 13+.

SYSTEM FEATURES

3.1 User Registration The system shall allow users to sign up using university email. Verification is done via an OTP sent to the registered phone number.
3.2 Ride Creation and Booking Drivers can post rides including departure, destination, available seats, and cost per seat. Passengers can search and book available rides. GPS-based route matching improves accuracy. System automatically matches drivers and passengers with similar routes.
3.3 Wallet and Payment System Users can deposit money using bank cards or PayPal. Refunds are processed within 7 days. The app does not retain financial data of users.

EXTERNAL INTERFACE REQUIREMENTS

4.1 Hardware Interfaces The app shall be optimized for mobile devices running Android 8.0+ only.
4.2 Software Interfaces Integration with PayPal API for payments. Integration with University LDAP for identity verification.
4.3 Communication Interfaces The application will communicate with its backend server using HTTPS. Push notifications will be sent via Firebase Cloud Messaging.

NON-FUNCTIONAL REQUIREMENTS

5.1 Security The system must comply with GDPR and must not store any personal data such as phone numbers, emails, or bank details. The system must authenticate users using their university email.
5.2 Performance The system should support 500 concurrent users with less than 1 second response time. Database updates may take up to 5 seconds to reflect in the user interface.
5.3 Reliability The system should have 99.9% uptime. System may be unavailable every Sunday from 2–6 AM for maintenance.
5.4 Usability The interface shall be simple and intuitive for both students and staff. All buttons, labels, and error messages will follow the university’s accessibility guidelines.

SYSTEM CONSTRAINTS

6.1 Regulatory Policies The app must comply with both GDPR and U.S. data privacy laws.
6.2 Design Limitations All data must be stored on local devices only (no cloud usage). Data backups must be automatically uploaded to cloud storage daily.
6.3 Business Rules No automated features should replace user decision-making. System must use AI route-matching to recommend rides automatically.

User Story 1 As a student, I want to access CampusRide through my laptop’s web browser so that I can plan my trips from my dorm room without using my phone.
User Story 2 As a driver, I want to receive passenger requests automatically matched to my route so that I don’t have to manually review every booking.
User Story 3 As a passenger, I want to search for rides using filters like departure time, destination, and driver rating so that I can choose the most suitable ride.
User Story 4 As a driver, I want to post my ride even if I haven’t verified my university email yet so that I can offer rides quickly after downloading the app.
User Story 5 As a user, I want to pay for my ride using cash so that I don’t need to link my bank card to the wallet.
User Story 6 As a staff member, I want to use the CampusRide web portal from my office computer so that I can manage and monitor ride-sharing activity among students.
User Story 7 As a passenger, I want to be able to cancel my ride and get an instant refund to my in-app wallet balance so that I can easily change my travel plans.
User Story 8 As a driver, I want CampusRide to automatically adjust my route based on live GPS tracking so that I can pick up passengers more efficiently.
User Story 9 As a user, I want to register without providing my phone number so that my privacy is protected.
User Story 10 As an administrator, I want to see reports that include users’ contact information and payment history so that I can verify transactions when needed.
User Story 11 As a user, I want to back up my data automatically to cloud storage so that I don’t lose my ride history if I change my phone.
User Story 12 As a security officer, I want the app to store all personal and trip data locally on the user’s device so that no information leaves the university network.
User Story 13 As a student, I want the app to send me push notifications for upcoming rides even when I’m offline so that I don’t miss my carpool schedule.
User Story 14 As a passenger, I want the app to let me join a ride even if the driver hasn’t confirmed it yet so that I can reserve my seat immediately.
User Story 15 As a user, I want CampusRide to be available in multiple languages so that international students can easily navigate the interface.
